# 2.5D Dual-Solver Simulation Plan

## Decision

The active physics plan starts at 2.5D. The previous 2D river/raft path is retired and should not drive future architecture.

The project will build two versions of the same 2.5D simulation:

- A Python/PyClaw reference model for physically credible shallow-water behavior.
- A custom C++ reduced shallow-water / height-field solver intended for the eventual Unreal runtime.

Both versions must run the same procedurally generated scenario. The C++ solver is tuned and accepted only by matching PyClaw reference outputs within defined tolerances.

## Goal

Create a deterministic, headless 2.5D modeling platform for white water rafting:

- Bed elevation
- Water surface elevation
- Depth
- Horizontal velocity
- Wet/dry state
- Hydraulic jumps and wave trains
- Holes, ledges, laterals, boils, shallows, and rocks
- Raft buoyancy, drag, grounding, wave climb, pitch, roll, surf, flush, pin, and flip outcomes

The Python model is the truth source during research. The C++ model is the runtime candidate.

## Definition Of 2.5D

Use a depth-averaged river domain:

- `x, y`: horizontal world position
- `z`: raft/body vertical position
- `bed(x, y)`: riverbed elevation
- `eta(x, y, t)`: water surface elevation
- `h(x, y, t) = eta - bed`: water depth
- `u, v`: depth-averaged horizontal velocity
- `hu, hv`: momentum
- `wet`: wet/dry cell state
- `normal`: local water-surface normal derived from `eta`
- `slope`: local surface gradient
- `source_terms`: bathymetry, roughness, inflow, obstacles, authored feature forcing
- `mu_eff`: effective damping or aeration coefficient for raft-force coupling

This is not full 3D CFD. It is a height-field / shallow-water model that produces physically interpretable water fields for raft interaction and eventual Unreal visualization.

## Shared Scenario Contract

The most important architectural rule: PyClaw and C++ run the same scenario.

Every scenario must be represented by a solver-neutral data package:

- Seed and generator version
- World bounds, grid resolution, timestep policy, and duration
- Bed elevation grid or procedural bed function
- Initial water surface/depth
- Initial momentum/velocity
- Boundary conditions and inflow hydrographs
- Manning/roughness or equivalent friction parameters
- Rocks, ledges, shallow shelves, constrictions, and strainers
- Authored feature metadata for holes, laterals, wave trains, and boils
- Raft physical parameters
- Probe points, cross sections, and comparison windows
- Expected telemetry channels

The scenario generator should produce:

- `scenario.json`: human-readable metadata
- `bed.npy` or equivalent binary grid
- `initial_state.npy`
- `features.json`
- `probes.json`
- Optional diagnostic PNGs

Both solvers load this package without divergent interpretation.

## PyClaw Reference Model

PyClaw is the Python reference implementation for 2.5D water modeling.

Responsibilities:

- Solve depth-averaged shallow-water equations for generated scenarios.
- Exercise finite-volume behavior, wave propagation, wet/dry handling, source terms, and boundary conditions.
- Produce reference fields for `h`, `eta`, `u`, `v`, `hu`, `hv`, Froude number, wet/dry masks, and derived surface normals.
- Export frame-by-frame water-field telemetry and probe data.
- Provide reference raft-force inputs for later coupling tests.

The first PyClaw phase should avoid overfitting to complex generated rapids. Start with canonical cases:

- Flat pool
- Uniform channel flow
- Dam-break / bore
- Bed step
- Constriction
- Wet/dry shoreline
- Standing-wave-like hydraulic transition

Then apply the procedural rafting scenarios.

Validation:

- Use SWASHES-style analytic cases where available.
- Check mass conservation.
- Check wet/dry behavior.
- Check hydraulic jump location and speed.
- Check bounded velocity and depth.
- Save probe and cross-section outputs for C++ comparison.

## Custom C++ Runtime Solver

The custom C++ solver is a reduced shallow-water / height-field model designed for real-time use.

Responsibilities:

- Load the same scenario package as PyClaw.
- Step water state with deterministic fixed timesteps.
- Provide `h`, `eta`, `u`, `v`, wet/dry state, surface normals, slope, and feature tags to raft and rendering systems.
- Run fast enough for eventual Unreal/VR budgets.
- Expose the same telemetry channels as the PyClaw reference.
- Support platform-specific runtime modes without changing scenario semantics.

Initial implementation direction:

- CPU first, with data-oriented arrays.
- Explicit finite-volume or reduced height-field update.
- Conservative handling for depth and momentum where practical.
- Stable wet/dry approximation.
- Roughness/friction source terms.
- Boundary condition support for inflow, outflow, walls, and banks.
- Optional authored feature forcing for holes, laterals, boils, and wave trains.

The C++ model does not need to match PyClaw internals. It must match PyClaw outputs and raft-relevant behavior.

## Solver Matching

Run both solvers against the same scenario and compare:

- `h`, `eta`, `u`, `v`, `hu`, `hv`
- Wet/dry masks
- Mass conservation
- Energy trends
- Froude number
- Hydraulic jump location
- Wave crest/trough location and amplitude
- Hole retention zone geometry
- Lateral wave direction and strength
- Boil/upwelling proxy region
- Raft force samples
- Raft trajectory and outcome classification once coupled

Metrics:

- L1/L2/Linf error on fields
- Probe time-series error
- Cross-section error
- Feature-location error
- Mass-conservation drift
- Runtime cost per simulated second
- Raft outcome agreement: clear, stall, surf, flush, ground, pin, flip

Tuning:

- Tune C++ numerical coefficients, friction, feature forcing, damping, and raft coupling parameters against PyClaw.
- Store tuned parameter sets by scenario class.
- Reject parameter changes that improve one feature while regressing canonical validation cases.

## Raft Coupling

The raft should sample either solver through the same interface:

- Water surface height
- Surface normal
- Depth
- Horizontal velocity
- Wet/dry state
- Bed height
- Roughness/damping
- Feature tags

The raft force model should be solver-neutral:

- Buoyancy from submerged sample depth and surface normal
- Vertical damping from water-relative velocity
- Horizontal drag from depth-averaged velocity
- Added-mass approximation
- Grounding contact against bed and submerged rocks
- Wave climb and crest impact
- Hole retention and surf/flush classification
- Paddle blade force based on submerged depth and local water velocity

When possible, the exact same raft probe set and force code should run against PyClaw exports and C++ runtime fields.

## Workflow

1. Generate one procedural scenario package.
2. Run PyClaw reference simulation.
3. Export PyClaw fields, probes, cross sections, and telemetry.
4. Run custom C++ solver on the same package.
5. Export C++ fields, probes, cross sections, and telemetry.
6. Compare field and raft-force metrics.
7. Tune C++ parameters.
8. Repeat until acceptance thresholds pass.
9. Promote the scenario into regression tests.

## Acceptance Criteria

The 2.5D solver program is viable when:

- PyClaw can run canonical and generated scenarios deterministically.
- The custom C++ solver can run the same scenarios from the same files.
- Field errors stay within defined tolerances for canonical cases.
- Raft force samples match PyClaw-derived references closely enough for gameplay-critical behavior.
- Scenario outcomes match: clear, stall, surf, flush, ground, pin, or flip.
- C++ runtime cost is inside the target budget for Unreal prototype use.
- Regressions are caught by automated tests and comparison reports.

## Milestones

### Milestone A: Retire 2D Path

- Remove 2D-first assumptions from docs and TODOs.
- Treat existing 2D code as legacy scratch code until it is deleted or replaced.
- Do not build new architecture on `river2d.py` or `raft2d.py`.

### Milestone B: Shared 2.5D Scenario Schema

- Define scenario metadata, grids, features, probes, and expected telemetry.
- Generate deterministic scenarios from seed.
- Export identical inputs for PyClaw and C++.
- Add fixture scenarios before procedural rapids.

### Milestone C: PyClaw Reference

- Integrate PyClaw as an optional research dependency.
- Implement canonical shallow-water fixtures.
- Export field frames, probes, and comparison artifacts.
- Validate against analytic/SWASHES-style cases where practical.

### Milestone D: Custom C++ Solver

- Create standalone C++ solver library/executable.
- Load shared scenario packages.
- Implement reduced shallow-water / height-field stepping.
- Export matching telemetry.
- Add C++ unit and regression tests.

### Milestone E: Comparison And Tuning

- Run both solvers on identical scenarios.
- Compare fields, probes, mass conservation, feature metrics, and raft forces.
- Tune C++ parameters to match PyClaw.
- Promote passing scenarios to regression fixtures.

### Milestone F: Raft Coupling

- Add 6-DoF raft sampling against solver-neutral water fields.
- Run raft against PyClaw outputs and C++ runtime fields.
- Compare force envelopes and outcome classifications.

### Milestone G: Unreal Readiness

- Keep PyClaw offline as the reference model.
- Use the custom C++ solver as the Unreal runtime candidate.
- Feed the same scenario and telemetry schema into Unreal replay/debug tools.
- Move to Unreal only after solver matching and profiling are complete.
