# 2.5D Raft Simulation Plan

## Goal

Move from the current top-down 2D analytic river model to a 2.5D simulator that gives the raft a water surface, bed elevation, vertical interaction forces, pitch, roll, wave climbing, hole retention, grounding, and physically inspectable buoyancy without jumping all the way to full 3D CFD.

The 2.5D engine should remain headless, deterministic, and testable in Python. It is the bridge between the current 2D force-field sandbox and the later Chrono/Unreal/VR runtime.

## Definition Of 2.5D

Use a horizontal river domain with height-aware fields:

- `x, y`: horizontal world position
- `z`: raft/body vertical position
- `eta(x, y, t)`: water surface elevation
- `bed(x, y)`: riverbed elevation
- `h(x, y, t) = eta - bed`: water depth
- `u, v`: horizontal water velocity
- `w`: vertical/upwelling velocity proxy
- `normal`: local water-surface normal
- `slope`: local surface gradient
- `mu_eff`: effective damping or aeration coefficient
- `turbulence`: deterministic disturbance intensity

This model is not full volumetric water. It treats the water column as a height field and uses sampled hull points or pontoons to estimate submerged depth, buoyancy, drag, impact, and contact.

## Why This Before Full 3D

2.5D gives the project the most important rafting behaviors earlier:

- Raft climbs and stalls on standing waves.
- The bow can pearl into a hole.
- Cross-current waves can lift one tube and roll the raft.
- Shallow shelves can ground the floor or one side tube.
- Boils and upwellings can pop the boat upward or unload steering.
- VR guide movement can eventually feel connected to pitch, roll, and paddle depth.

It also keeps validation practical. We can render diagnostic plots, run pytest scenarios, compare against analytic shallow-water cases, and fit coefficients before native Chrono/Unreal integration.

## Data Model

### River Surface Field

Add a `River25D` or equivalent module after the current `River2D` stabilizes.

The first data structure should expose:

- Centerline and cross-section queries inherited from the 2D generator.
- Bed elevation sampled along and across the river.
- Surface elevation sampled from procedural waves and later solver output.
- Depth, velocity, surface normal, slope, turbulence, damping, and feature tags.
- Rock/ledge collision geometry with exposed dry height and submerged height.
- Deterministic time sampling for moving waves and boils.

The first implementation can layer 2.5D fields on top of `GeneratedRiver2D`:

1. Generate bed profile from gradient, constrictions, and features.
2. Generate mean water surface from flow volume and bed profile.
3. Add analytic standing waves, wave trains, holes, laterals, and boils as height-field perturbations.
4. Derive normals and slope from finite differences.
5. Keep the current 2D flow vectors as the horizontal velocity field.

### Raft Body State

Upgrade from planar 2D state to a 6-DoF rigid body approximation:

- Position `(x, y, z)`
- Orientation quaternion
- Linear velocity `(vx, vy, vz)`
- Angular velocity `(wx, wy, wz)`
- Mass and inertia tensor
- Sampled buoyancy/contact points around tubes and floor
- Optional guide/passenger mass offsets

Keep a simplified mode available:

- 3-DoF planar raft for fast 2D tests
- 6-DoF rigid raft over 2.5D water for vertical dynamics
- Later rigid raft plus compliant shell

### Hull Sampling

Represent the first 2.5D raft as sampled points and small patches:

- Bow tube samples
- Stern tube samples
- Port/starboard tube samples
- Floor samples
- Paddle blade samples
- Optional passenger/guide mass points

Each sample stores:

- Local position on raft
- Effective area
- Buoyancy coefficient
- Drag coefficient
- Contact softness
- Tube/floor/hard-point tag

This keeps the model inspectable before a mesh or XPBD raft shell exists.

## Force Model

### Buoyancy

For each hull sample:

1. Transform the local sample to world space.
2. Sample water surface elevation and normal.
3. Compute submerged depth or signed distance to the surface.
4. Apply upward force along the surface normal.
5. Clamp force by local displaced volume approximation.
6. Add damping from vertical relative velocity.

Telemetry:

- Sample depth
- Buoyancy force
- Surface normal
- Vertical damping
- Total buoyant impulse per tube/floor region

### Gravity And Weight Distribution

Apply gravity at the raft center of mass and optional crew/passenger mass points. This is needed for realistic pitch and roll, especially when a guide stands in the stern or when the bow climbs a wave.

Telemetry:

- Center of mass
- Crew mass offsets
- Weight torque

### Surface Slope And Wave Climb

Standing waves and wave trains become actual surface height barriers:

- Bow samples climb the upstream face.
- Gravity and buoyancy determine whether the boat clears or stalls.
- Crest impacts apply vertical and pitch impulses.
- Downstream troughs can slam the floor.

Telemetry:

- Crest crossing
- Wave potential estimate
- Bow/stern immersion
- Stall/surf/flush classification

### Holes And Hydraulics

A 2.5D hole combines:

- Surface depression upstream or at the boil line
- Retentive horizontal velocity field
- Upstream-facing surface slope
- Aerated low-damping region
- Downstream boil/upwelling patch

Telemetry:

- Retention force
- Bow pearl depth
- Stern lift
- Surf duration
- Escape vector

### Eddy Lines And Shear

Carry the 2D shear model forward, but add vertical and roll coupling:

- One tube in slow/reverse water and the other in fast current creates yaw and roll torque.
- Strong eddy lines can dip one side tube and lift the other.
- Aerated or turbulent seams reduce damping and paddle authority.

Telemetry:

- Velocity gradient at each hull sample
- Roll torque
- Yaw torque
- Damping modifier

### Rocks, Ledges, And Grounding

Rocks and bed features need height:

- Exposed rocks collide with tubes.
- Submerged rocks can scrape or launch the floor.
- Ledges create bed steps and downstream holes.
- Shallow shelves apply grounding friction when floor/tube samples penetrate bed.

Telemetry:

- Contact normal
- Penetration depth
- Tube/floor tag
- Contact impulse
- Grounding drag
- Pinning score

### Paddle And Guide Input

Paddle blades now sample depth:

- Blade force only applies when submerged.
- Power depends on blade velocity relative to local water.
- Draws, pries, sweeps, and braces can create roll and yaw torque.
- VR controller pose can later drive blade path and depth.

Telemetry:

- Blade depth
- Blade-water relative velocity
- Stroke force
- Stroke torque
- Missed/air stroke detection

## Numerical Architecture

Initial 2.5D loop:

1. Advance procedural time or water solver state.
2. Predict raft transform.
3. Sample hull points against surface, velocity, bed, rocks, and hazards.
4. Accumulate gravity, buoyancy, damping, drag, wave, contact, and paddle forces.
5. Integrate 6-DoF rigid body state using a fixed timestep.
6. Resolve hard rock/bed contacts with substeps when needed.
7. Classify outcomes: clear, stalled, surfed, flushed, pinned, grounded, flipped.
8. Record force, torque, surface, and hull sample telemetry.

Start with semi-implicit Euler plus substepping. Move to Chrono rigid-body integration once the reduced-force model is validated, then migrate the same tests to native Chrono.

## Validation Scenarios

Create named pytest scenarios before broad procedural generation:

- Flat pool float: raft settles to stable draft with no horizontal drift.
- Calm current: raft drifts downstream while maintaining bounded pitch/roll.
- Single standing wave: clear at sufficient speed, stall/surf below threshold.
- Wave train: repeated pitch impulses remain bounded.
- Hole surf: retention holds raft until escape stroke or sufficient velocity.
- Eddy-line peel-out: yaw and roll torque appear when only part of raft crosses.
- Lateral wave: sideways lift and roll impulse match expected direction.
- Shallow shelf: grounding drag increases and raft can pivot on one tube.
- Submerged rock: floor/tube contact creates upward and lateral impulse.
- Boil/upwelling: vertical impulse and damping reduction are deterministic.

Each scenario should assert:

- Determinism for fixed seed and timestep.
- Bounded energy in passive cases.
- Correct sign and rough magnitude of expected force/torque.
- Expected outcome classification.
- No unbounded spin, launch, or sinking.

## Milestone Breakdown

### Milestone A: 2.5D Field Layer

- Add `River25D` query API.
- Derive bed, surface, depth, slope, and normals from 2D rivers.
- Add analytic wave, hole, ledge, lateral, boil, and shallow height fields.
- Add surface/bed plotting and section plotting.
- Add validation for bounded depth, slope, and surface continuity.

### Milestone B: 6-DoF Rigid Raft

- Add 3D state with quaternion orientation.
- Add inertia tensor and rigid body integration.
- Add hull sample patches for tubes and floor.
- Add gravity, buoyancy, vertical damping, and water drag.
- Add draft/trim stability tests.

### Milestone C: 2.5D River Feature Dynamics

- Convert standing waves from planar barriers to surface-height barriers.
- Convert holes to depression/retention/upwelling systems.
- Convert lateral waves to height plus side impulse fields.
- Add bed and rock height contact.
- Add shallow grounding and pinning detection.

### Milestone D: Paddle And VR-Ready Interaction

- Add blade pose, depth, and water-relative force model.
- Add scripted guide strokes for validation.
- Define future VR input mapping for paddle shaft/blade pose.
- Export paddle telemetry suitable for replay in Unreal.

### Milestone E: Chrono Migration Path

- Keep the Python 2.5D solver as the reference model.
- Add a Chrono rigid-body adapter for 6-DoF raft motion.
- Compare Python and Chrono integration on identical force logs.
- Define which force fields remain Python-authored and which move to C++.

## Implementation Order

1. Add 2.5D data classes and a query wrapper around the existing 2D river.
2. Build surface/bed diagnostic plots before adding raft dynamics.
3. Add a static raft draft test in a flat pool.
4. Add 6-DoF rigid-body integration and telemetry.
5. Add standing-wave height barriers and one wave-clear/stall test.
6. Add hole and shallow-shelf tests.
7. Add Chrono 6-DoF adapter after Python behavior is stable.

The guiding rule is to keep every 2.5D effect visible in telemetry and tests before it becomes part of the full game runtime.
