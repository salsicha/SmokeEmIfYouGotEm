# 2.5D Dual-Solver Simulation Plan

## Decision

The active physics plan starts at 2.5D. The previous 2D river/raft path is retired and should not drive future architecture.

The project will build two versions of the same 2.5D simulation:

- A GeoClaw reference model for physically credible shallow-water/geophysical-flow behavior.
- A custom C++ reduced shallow-water / height-field solver intended for the eventual Unreal runtime.

Both versions must run the same procedurally generated or real-world geospatial scenario. The C++ solver is tuned and accepted only by matching GeoClaw reference outputs within defined tolerances. The detailed acceptance gate is tracked in [Custom C++ Engine Full Validation Plan](custom-cpp-engine-validation-plan.md).

## Goal

Create a deterministic, headless 2.5D modeling platform for white water rafting:

- Bed elevation
- Water surface elevation
- Depth
- Horizontal velocity
- Wet/dry state
- Hydraulic jumps and wave trains
- Holes, ledges, laterals, boils, shallows, and rocks
- Real-world river course/elevation data, rapid annotations, seasonal flow presets, and difficulty-driven fluid parameters
- Raft buoyancy, drag, grounding, wave climb, pitch, roll, surf, flush, pin, and flip outcomes

The Python model is the truth source during research. The C++ model is the runtime candidate.

## Definition Of 2.5D

Use a depth-averaged river domain:

- `x, y`: horizontal world position
- `z`: raft/body vertical position
- `bed(x, y)`: riverbed elevation
- `eta(x, y, t)`: water surface elevation
- `h(x, y, t) = eta - bed`: water depth
- `u, v`: depth-averaged horizontal velocity
- `hu, hv`: momentum
- `wet`: wet/dry cell state
- `normal`: local water-surface normal derived from `eta`
- `slope`: local surface gradient
- `source_terms`: bathymetry, roughness, inflow, obstacles, authored feature forcing
- `mu_eff`: effective damping or aeration coefficient for raft-force coupling

This is not full 3D CFD. It is a height-field / shallow-water model that produces physically interpretable water fields for raft interaction and eventual Unreal visualization.

## Shared Scenario Contract

The most important architectural rule: GeoClaw and C++ run the same scenario.

Every scenario must be represented by a solver-neutral data package:

- Seed and generator version
- Scenario type: fixture, procedural, or real-world
- Source manifest, river id, section id, coordinate reference system, processing version, and confidence scores for real-world sections
- World bounds, grid resolution, timestep policy, and duration
- Bed elevation grid or procedural bed function
- Initial water surface/depth
- Initial momentum/velocity
- Boundary conditions and inflow hydrographs
- Manning/roughness or equivalent friction parameters
- Rocks, ledges, shallow shelves, constrictions, and strainers
- Authored feature metadata for holes, laterals, wave trains, and boils
- Reviewed rapid annotations, gauge metadata, season preset, flow percentile/band, difficulty preset, and adaptive fluid-parameter set where available
- Raft physical parameters
- Probe points, cross sections, and comparison windows
- Expected telemetry channels

The scenario generator should produce:

- `scenario.json`: human-readable metadata
- `bed.npy` or equivalent binary grid
- `initial_state.npy`
- `features.json`
- `probes.json`
- Optional `source_manifest.json`, `centerline.geojson`, `cross_sections.geojson`, `rapid_annotations.geojson`, `flow_presets.json`, and `confidence.json` for real-world scenarios
- Optional diagnostic PNGs

Both solvers load this package without divergent interpretation. The GeoClaw exporter may create GeoClaw-specific run/topography/fixed-grid-output files, but those files must be generated from the shared package.

## Variable Cascading Scenario Package

The next scenario-package revision should model a playable river as a sequence of connected 2.5D reaches rather than one mostly uniform channel. This is the right direction for California pool-and-drop rivers: long lower-gradient pools and recovery zones are interrupted by short steeper drops, constrictions, boulder gardens, ledges, wave trains, and hydraulic controls.

The design goal is to make that structure explicit while keeping one solver-neutral source of truth. A cascading package should add:

- `reaches`: ordered downstream sections with station range, local grid transform, slope profile, width profile, bank shape, roughness, boulder density, vegetation/debris flags, and confidence metadata.
- `drop_transitions`: hydraulic connectors between reaches with crest station, bed-elevation fall, ramp or ledge length, tailwater depth, expected hydraulic control, aeration/turbulence proxy, recirculation risk, eddy recovery zones, and hazard tags.
- `pools`: low-gradient storage/recovery sections with depth, eddy, recirculation, and tailwater controls. Pools should remain active hydraulic features, not blank space between rapids.
- `stitching`: either one global grid annotated by reach/drop IDs, multiple reach-local grids with overlap/ghost zones, or both. GeoClaw and C++ must consume equivalent bathymetry, initial state, boundaries, and roughness regardless of storage layout.
- `handoff diagnostics`: mass flux, momentum flux, surface-elevation continuity where physically expected, energy loss across drops, wet/dry front behavior, and raft-state continuity when crossing reach boundaries.

The important caveat: drops should be geometry and boundary conditions, not scripted boosts or teleports. Some energy loss across ledges, holes, and turbulent drops is physically expected, but it should be represented through bed topography, roughness, hydraulic controls, aeration/damping proxies, and validated source terms that can be compared against GeoClaw.

This package shape should make the South Fork American baseline more useful. A Chili Bar to Coloma seed can be represented as alternating pools, tongues, drops, boulder gardens, wave trains, eddies, and recovery pools while still exporting one comparable GeoClaw/C++ validation run.

## GeoClaw Reference Model

GeoClaw is the offline reference implementation for 2.5D river-water modeling.

Responsibilities:

- Solve depth-averaged shallow-water equations for fixture, procedural, and real-world scenarios.
- Exercise finite-volume behavior, wave propagation, wet/dry handling, topography/bathymetry source terms, roughness/friction, adaptive mesh refinement, and boundary conditions.
- Produce reference fields for `h`, `eta`, `u`, `v`, `hu`, `hv`, Froude number, wet/dry masks, and derived surface normals.
- Export frame-by-frame water-field telemetry and probe data.
- Provide reference raft-force inputs for later coupling tests.

The first GeoClaw phase should avoid overfitting to complex generated rapids. Start with canonical cases:

- Flat pool
- Uniform channel flow
- Dam-break / bore
- Bed step
- Constriction
- Wet/dry shoreline
- Sloping channel with Manning friction
- Drop/ledge over variable topography
- Standing-wave-like hydraulic transition

Then apply the procedural rafting scenarios.

Validation:

- Use SWASHES-style analytic cases where available.
- Check mass conservation.
- Check wet/dry behavior.
- Check hydraulic jump location and speed.
- Check bounded velocity and depth.
- Save probe and cross-section outputs for C++ comparison.

PyClaw artifacts from earlier milestones remain legacy regression data, but GeoClaw is the new acceptance reference.

## Custom C++ Runtime Solver

The custom C++ solver is a reduced shallow-water / height-field model designed for real-time use.

Responsibilities:

- Load the same scenario package as GeoClaw.
- Step water state with deterministic fixed timesteps.
- Provide `h`, `eta`, `u`, `v`, wet/dry state, surface normals, slope, and feature tags to raft and rendering systems.
- Run fast enough for eventual Unreal/VR budgets.
- Expose the same telemetry channels as the GeoClaw reference.
- Support platform-specific runtime modes without changing scenario semantics.

Initial implementation direction:

- CPU first, with data-oriented arrays.
- Explicit finite-volume or reduced height-field update.
- Conservative handling for depth and momentum where practical.
- Stable wet/dry approximation.
- Roughness/friction source terms.
- Boundary condition support for inflow, outflow, walls, and banks.
- Optional authored feature forcing for holes, laterals, boils, and wave trains.

The C++ model does not need to match GeoClaw internals. It must match GeoClaw outputs and raft-relevant behavior. Fallback or initial-state-only GeoClaw normalization is useful for schema smoke tests, but it does not count as full shallow-water solution validation.

Current Milestone 3 artifact:

- `physics/cpp/` contains the standalone C++17 solver library and CMake build.
- `raftsim_water_solver` loads a shared package and writes comparison-ready CSV/JSON outputs.
- `raftsim_water_tests` validates package loading, deterministic stepping, native output writing, and basic stability.
- The first reduced update includes bed-gradient acceleration through `eta`, Manning-style roughness damping, boundary handling, wet/dry clamping, and authored forcing for holes, laterals, boils, wave trains, ledges, and shallows.
- The model is not accepted as final physics until the comparison harness tunes it against GeoClaw reference runs.

## Solver Matching

Run both solvers against the same scenario and compare:

- `h`, `eta`, `u`, `v`, `hu`, `hv`
- Wet/dry masks
- Mass conservation
- Energy trends
- Froude number
- Hydraulic jump location
- Wave crest/trough location and amplitude
- Hole retention zone geometry
- Lateral wave direction and strength
- Boil/upwelling proxy region
- Raft force samples
- Raft trajectory and outcome classification once coupled

Metrics:

- L1/L2/Linf error on fields
- Probe time-series error
- Cross-section error
- Feature-location error
- Mass-conservation drift
- Runtime cost per simulated second
- Raft outcome agreement: clear, stall, surf, flush, ground, pin, flip

Tuning:

- Tune C++ numerical coefficients, friction, feature forcing, damping, and raft coupling parameters against GeoClaw.
- Store tuned parameter sets by scenario class.
- Reject parameter changes that improve one feature while regressing canonical validation cases.

## Raft Coupling

The raft should sample either solver through the same interface:

- Water surface height
- Surface normal
- Depth
- Horizontal velocity
- Wet/dry state
- Bed height
- Roughness/damping
- Feature tags

The raft force model should be solver-neutral:

- Buoyancy from submerged sample depth and surface normal
- Vertical damping from water-relative velocity
- Horizontal drag from depth-averaged velocity
- Added-mass approximation
- Grounding contact against bed and submerged rocks
- Wave climb and crest impact
- Hole retention and surf/flush classification
- Paddle blade force based on submerged depth and local water velocity

When possible, the exact same raft probe set and force code should run against GeoClaw exports and C++ runtime fields.

## Workflow

1. Generate one procedural scenario package.
2. Run GeoClaw reference simulation.
3. Export GeoClaw fixed-grid fields, probes, cross sections, and telemetry.
4. Run custom C++ solver on the same package.
5. Export C++ fields, probes, cross sections, and telemetry.
6. Compare field and raft-force metrics.
7. Tune C++ parameters.
8. Repeat until acceptance thresholds pass.
9. Promote the scenario into regression tests.
10. For real-world sections, repeat the comparison across low, median, and high runnable seasonal flow presets before approving the scenario for Unreal preproduction.
11. For cascading river packages, validate both per-reach output and stitched whole-window output so section boundaries do not hide water-field or raft-transition errors.

## Acceptance Criteria

The 2.5D solver program is viable when:

- GeoClaw can run canonical, procedural, and real-world scenarios deterministically.
- GeoClaw validation uses full fixed-grid solution frames rather than setup-only or initial-state-only fallback exports.
- The custom C++ solver can run the same scenarios from the same files.
- Field errors stay within defined tolerances for canonical cases.
- Raft force samples match GeoClaw-derived references closely enough for gameplay-critical behavior.
- Scenario outcomes match: clear, stall, surf, flush, ground, pin, or flip.
- C++ runtime cost is inside the target budget for Unreal prototype use.
- Real-world river scenario packages preserve source provenance, geospatial transforms, season/flow/difficulty choices, and confidence metadata while producing comparable GeoClaw and C++ outcomes.
- Regressions are caught by automated tests and comparison reports.

## Milestones

### Milestone A: Retire 2D Path

- Remove 2D-first assumptions from docs and TODOs.
- Treat existing 2D code as legacy scratch code until it is deleted or replaced.
- Do not build new architecture on `river2d.py` or `raft2d.py`.

### Milestone B: Shared 2.5D Scenario Schema

- Define scenario metadata, grids, features, probes, and expected telemetry.
- Generate deterministic scenarios from seed.
- Export identical semantic inputs for GeoClaw and C++; GeoClaw-specific files are generated from the shared package.
- Add fixture scenarios before procedural rapids.

### Milestone C: GeoClaw Reference

- Integrate GeoClaw as an optional research dependency.
- Implement canonical shallow-water fixtures.
- Export field frames, probes, and comparison artifacts.
- Validate against analytic/SWASHES-style cases where practical.

### Milestone D: Custom C++ Solver

- Create standalone C++ solver library/executable.
- Load shared scenario packages.
- Implement reduced shallow-water / height-field stepping.
- Export matching telemetry.
- Add C++ unit and regression tests.

### Milestone E: Comparison And Tuning

- Run both solvers on identical scenarios.
- Compare fields, probes, mass conservation, feature metrics, and raft forces.
- Tune C++ parameters to match GeoClaw.
- Promote passing scenarios to regression fixtures.

### Milestone F: Raft Coupling

- Add 6-DoF raft sampling against solver-neutral water fields.
- Run raft against GeoClaw outputs and C++ runtime fields.
- Compare force envelopes and outcome classifications.

### Milestone G: Unreal Readiness

- Keep GeoClaw offline as the reference model.
- Use the custom C++ solver as the Unreal runtime candidate.
- Feed the same scenario and telemetry schema into Unreal replay/debug tools.
- Move to Unreal only after solver matching, profiling, and at least one validated real-world river/season/difficulty scenario package are complete.

### Milestone H: Real-World Scenario Packages

- Convert selected river sections from geospatial course/elevation data, reviewed imagery/rapid annotations, and seasonal flow research.
- Encode gauge metadata, flow percentile bands, season presets, difficulty presets, and adaptive fluid parameters.
- Run each approved section through GeoClaw and the custom C++ solver on shared inputs.
- Export Unreal-ready corridor metadata only after low, median, and high runnable presets pass comparison thresholds.

### Milestone I: Variable Cascading River Packages

- Replace monolithic procedural and real-world packages with ordered reach/drop sequences.
- Generate California-style pool-and-drop chains with variable slopes, constrictions, ledges, wave trains, eddies, and recovery pools.
- Add package-level and reach-level probes, cross sections, comparison windows, and raft entry/exit checkpoints.
- Validate GeoClaw output across stitched reach boundaries, sudden drops, pools, and hydraulic controls.
- Extend the C++ loader and tuning harness to match GeoClaw on the same cascading package before Unreal live-water work depends on it.
